关键输入

  • $_FILES['file']: 上传 PDF/HTML
  • $_POST['sender']: 第二步回传的文件路径
  • $_POST['limit']: 最大打开次数
  • $_POST['password']: 有效天数

关键输出

  • offline/{identifier}.csv: 计数与到期控制
  • {name}.html: 受保护阅读页（含PDF前半段）
  • {name}.zip: 最终下载物

我的逻辑解读（你要的“仔细看”总结）

1) DRM 实现不是“只靠 PDF 密码”，而是“HTML 包装 + 远程开锁”

drm.php 先用 mPDF 给原 PDF 加同一组密码，再把 PDF 的 base64 分成两段。 前段写进 HTML，后 399 字符写进 CSV。阅读时页面请求 yanhuo.php 获取后段，拼回完整数据。

2) 阅读次数与到期，核心在 yanhuo.php 而不在浏览器

每次打开 HTML 都会请求 yanhuo.php?e=year*month*day*id。 后端读取 CSV 后会先检查到期，再把次数 -1 写回 CSV，再决定返回后段字符串或 bad。

3) newpart3.txt 当前是“历史残留”

在 drm.php 里会读取 newpart3.txt，但写入代码被注释。 现在最终 HTML 只写 part0 + newpart1.txt。 不过 newpart1.txt 内已经内嵌了等价的拼接与校验逻辑，所以功能还能跑。

风险点与不一致项（已写上）

高风险: sender 路径信任过强

第二步直接使用 $_POST['sender'] 拼接路径读取文件。 缺少严格白名单校验，存在路径操纵风险。

中风险: “离线文件”依赖线上接口

生成 HTML 需要请求 https://maipdf.com/yanhuo.php 才能拿到 pdfleft。 网络不可用时无法正常阅读。

中风险: 过期返回值不统一

yanhuo.php 到期时有 bader，而前端主判断是 bad。 行为可能出现歧义。

低风险: 前后端限制不一致

前端 Dropzone 允许图片类型，但后端只接受 pdf/htm/html。 还有大小文案（90MB）与实际限制（222MB/约112MB）不一致。

风险总览表

问题 级别 影响 建议
sender 未严格校验 高 可读写路径被操纵 限制为固定目录 + basename 白名单
“离线”依赖 yanhuo.php 中 断网无法打开 明确文案为“联网校验阅读”
bader / bad 不统一 中 前端判断混乱 统一返回码，统一前端分支
前后端校验规则不一致 低 用户体验混乱 统一扩展名和大小阈值
