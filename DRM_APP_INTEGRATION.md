# DRM Secure Share App Integration

This Flutter app integrates with the MaiPDF Secure Share backend used by
`drm.maipdf.com`.

## What This Feature Does

The user selects a PDF in the app, chooses DRM rules, and receives a protected
`.maipdf` package. The app now does the PDF encryption and package generation
locally. The backend only stores the license row and the server-side key half.
The app adds a small `MAIPDF1` header and saves the result with a `.maipdf`
extension, so the package is opened through the MaiPDF protected viewer. Each
unlock is still checked by the DRM backend.

The long-term app goal is not only packaging. The app should also become a
protected reader: users open a protected `.maipdf`/HTML inside the app, read the PDF
there, and the reading screen enables platform screenshot/screen-recording
protection where the OS allows it.

This is different from the Cloud Sharing feature:

- Cloud Sharing creates an online MaiPDF share link.
- DRM Secure Share creates an encrypted offline `.maipdf` package.

## Pack API

Endpoint:

```text
POST https://drm.maipdf.com/api/app/licenses/create
```

Request type:

```text
application/json
```

Fields:

| Field | Required | Description |
|---|---:|---|
| `filename` | yes | Original PDF filename. |
| `file_size_bytes` | yes | Original PDF size. |
| `content_sha256` | yes | Base64url SHA-256 of the original PDF. |
| `keyshare_server` | yes | Base64url 32-byte server half generated by the app. |
| `max_opens` | no | Max unlock/open count. Default is `5`; `-1` means unlimited. |
| `expires_in_seconds` | no | Expiration from now in seconds. Default is `604800`; `-1` means never expires. |
| `chunk_count` | no | Currently `1`. |

Successful response:

- Status: `200`
- Body: JSON with `license_id`, `mod_code`, `api_base`, `created_at`
- Content-Type: `application/json`

Important response fields:

| JSON field | Description |
|---|---|
| `license_id` | Reading/license code. Readers use this when opening the locked HTML. |
| `mod_code` | Modification code. The owner must keep this to manage/revoke the package. |
| `api_base` | Usually `https://drm.maipdf.com`; the local HTML uses it for `/api/unlock`. |
| `created_at` | Unix timestamp embedded into the local package. |

## App Behavior

The Flutter page is `lib/screens/drm_secure_share_page.dart`.

Current flow:

1. User picks a PDF.
2. App locally generates AES-GCM key material and splits it into `html_half`
   and `server_half`.
3. App sends only license metadata and `server_half` to
   `/api/app/licenses/create`.
4. App locally encrypts the PDF, renders the locked HTML, builds a ZIP-compatible
   package, wraps it with the `MAIPDF1` header, and saves it as `.maipdf`:
   - Web: browser download.
   - Android/iOS: platform save flow through `file_picker`.
   - Windows/macOS/Linux: save dialog, then app writes the package bytes.
5. App displays `License ID` and `Modification Code`.
6. User can open the web management page:

```text
https://drm.maipdf.com/manage
```

## Protected Reader Goal

The intended reader-side flow is:

1. User receives a `.maipdf` file or locked HTML.
2. User opens/imports it in the MaiPDF app.
3. App unlocks through `https://drm.maipdf.com/api/unlock`.
4. App displays the PDF in a protected reading screen.
5. The protected reading screen enables anti-screenshot behavior.

Current viewer implementation:

- Screen: `lib/screens/drm_protected_viewer_page.dart`
- Home entry: DRM tab Ôćĺ `Protected Viewer`
- Supported input:
  - MaiPDF `.maipdf` generated locally by the app
  - Older MaiPDF DRM ZIP files for compatibility
  - Extracted locked `.html` file
- Package support is intentionally small: it reads the single STORED HTML entry
  generated by the app or by the older DRM backend. If a user re-compresses the
  HTML with a normal compressed ZIP tool, they can extract it first and choose
  the HTML.
- Rendering uses `webview_flutter` and loads the locked HTML in-app.

Platform notes:

- Android: use `WindowManager.LayoutParams.FLAG_SECURE` to block screenshots,
  recent-app previews, and most screen recording in the protected screen.
- iOS: Apple does not provide a normal public API that fully blocks screenshots.
  The app can detect screenshots after they happen, detect active screen
  recording/AirPlay, and hide sensitive content while capture is active.
- No app can stop someone from taking a photo of the phone with another camera.
  Watermarking, access limits, and audit logs are still needed.

Current native hook:

- Android method channel: `maipdf/secure_screen`
- Dart helper: `lib/utils/secure_screen.dart`

Usage for a future reader page:

```dart
@override
void initState() {
  super.initState();
  SecureScreen.setEnabled(true);
}

@override
void dispose() {
  SecureScreen.setEnabled(false);
  super.dispose();
}
```

## Important Product Notes

- The modification code cannot be recovered if lost.
- The protected `.maipdf` file is the file users send to recipients.
- The HTML can be distributed by email, chat, cloud drive, etc.
- The reader needs network access when unlocking/opening the protected HTML.
- This app should keep the DRM API contract local, so future App work does not
  depend on reading the `AustraliaGuide` source tree.
- In this first local-pack version, `.maipdf` is an app-facing container wrapper.
  It adds a `MAIPDF1` header before ZIP-compatible package bytes so normal ZIP
  tools do not open it directly. The PDF bytes inside the locked HTML are
  AES-GCM encrypted locally by the app, and access remains controlled by the DRM
  unlock flow. A future version can encrypt the outer container too if stronger
  file-format hiding is needed.
